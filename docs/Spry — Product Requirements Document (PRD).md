# 1. Project Overview

**Spry** is an open-source CLI tool that helps React Native (Expo) developers scaffold production-ready mobile apps and features using **Clean Architecture / DDD** principles. The developer defines a typed contract (abstract repository class with TypeScript decorators), and Spry generates all the boilerplate — data layer, use cases, state management, dependency injection — automatically.

- **npm package:** `spry` (confirmed available)
- **Companion package:** `@spry/decorators`
- **License:** MIT or Apache 2.0 (TBD)
- **Target platform:** React Native (Expo) — **mobile-only**, no web/RSC concerns
- **Stack:** Node.js CLI written in TypeScript

## Core Philosophy

> "Define once, generate everything."
> 

The developer writes the contract. Spry handles the mechanical boilerplate. This enforces Clean Architecture by default — the developer cannot accidentally couple layers because Spry generated the boundaries.

## Two Main Phases

1. `spry init` — project scaffolding, dependency installation, config
2. `spry new` + `spry build` — feature generation from domain contracts

# 3. Architecture Generated by Spry

Every feature Spry generates follows this folder structure:

```
src/
  features/
    <featureName>/
      presentation/
        views/           ← full screens (Expo Router pages import from here)
        components/      ← feature-scoped UI components
        hooks/           ← useFeature() composing React Query + Zustand
        state/
          <feature>Queries.ts    ← React Query hooks (server state)
          <feature>Store.ts      ← Zustand store (UI/client state only)
      domain/
        usecases/        ← one use case class per repository method
        repositories/    ← abstract repository class (developer writes this)
        models/          ← domain types (pure TypeScript, zero external imports)
        errors/          ← feature-specific DomainError subclasses + error codes
      data/
        repositories/    ← RepositoryImpl (concrete, calls datasource)
        datasources/     ← RemoteDataSource (HTTP calls)
        models/          ← DTOs (raw API shape)
      di.ts              ← dependency wiring, exports pre-built use case singletons
  shared/
    components/
    hooks/
    utils/
    constants/
    types/
    errors/              ← DomainError, HttpError, NetworkError (generated once)
    http/                ← httpClient.ts (fetch or axios wrapper)
```

## Layer Rules — strictly enforced by architecture

- **Domain layer** — zero external imports. Only TypeScript. Never knows about Axios, React Query, Zustand.
- **Data layer** — absorbs all HTTP differences. Swap fetch for axios? Only datasource changes.
- **Presentation layer** — absorbs all state management differences. Swap Zustand for Jotai? Only `state/` changes.
- **DTO → Domain mapping** — happens inside `data/repositories/`. Domain model never knows the API shape.

## Data Flow (always client-side — no RSC)

```
View → Hook → UseCase → RepositoryImpl → DataSource → HTTP
```

# 4. CLI Commands

## `spry init`

Initializes a new Expo project or configures an existing one. Prompts:

- **State management:** Zustand (Phase 1) / Jotai (Phase 2)
- **Network layer:** REST with fetch / REST with axios
- **Query client:** React Query (Phase 1) / urql (Phase 2)
- **API Routes:** Yes / No — enables Expo Router `+api.ts` endpoints for auth/BFF
- **Testing setup:** Vitest / none

After prompts:

1. Installs dependencies (detects npm / yarn / pnpm)
2. Patches `tsconfig.json` — adds `experimentalDecorators: true` and `emitDecoratorMetadata: true`
3. Generates `shared/errors/`, `shared/http/` scaffold
4. Writes `.spryrc.json` with a checksum
5. Writes `.spry-manifest.json` (empty)

## `spry new <featureName>`

Creates the domain skeleton for a new feature. Generates:

- `domain/repositories/<Feature>Repository.ts` — abstract class shell (developer fills in methods + decorators)
- `domain/models/<Entity>.ts` — type placeholder
- All other folders as empty directories with `.gitkeep`

Developer then:

1. Fills in the abstract class method signatures with `@spry/decorators`
2. Runs `spry build <featureName>`

## `spry build <featureName | all>`

Reads the abstract repository, parses it with ts-morph + Type Checker, generates all implementation files.

Flags:

- `--dry-run` — prints what would be generated/injected without writing anything
- `--force` — regenerates all Spry-owned files from scratch, asks for confirmation

# 5. `@spry/decorators` Package

Installed in the **Expo project** (not the CLI). Provides TypeScript decorators that the developer uses to annotate the abstract repository. The CLI reads these decorators statically via ts-morph AST — no runtime execution needed.

## Full API

```tsx
// Method decorators — HTTP verb + path
@GET(path: string)
@POST(path: string)
@PATCH(path: string)
@PUT(path: string)
@DELETE(path: string)

// Parameter decorators
@Param(name: string)    // path param :name
@Query(name: string)    // ?name=value
@Body()                 // request body
@Header(name: string)   // request header

// Class decorator
@BaseURL(url: string)   // base path for all methods

// Extra method decorators
@Cache(seconds: number) // generates staleTime in React Query
@Paginated()            // generates useInfiniteQuery instead of useQuery
```

## Example usage

```tsx
import { GET, PATCH, DELETE, Param, Body, BaseURL, Cache, Paginated } from '@spry/decorators'

@BaseURL('/api/v1')
export abstract class ProfileRepository {

  @GET('/profile/:userId')
  @Cache(60)
  abstract getProfile(@Param('userId') userId: string): Promise<UserProfile>

  @GET('/profiles')
  @Paginated()
  abstract getProfiles(@Query('page') page: number): Promise<PaginatedResult<UserProfile>>

  @PATCH('/profile/:userId')
  abstract updateProfile(
    @Param('userId') userId: string,
    @Body() input: UpdateProfileInput
  ): Promise<UserProfile>

  @DELETE('/profile/:userId')
  abstract deleteProfile(@Param('userId') userId: string): Promise<void>
}
```

# 6. TypeScript Parsing — How Spry Reads the Abstract Class

Spry uses **ts-morph** with the **TypeScript Type Checker** (not just AST). The Type Checker fully resolves types — follows imports, expands generics, detects nullability. This is the same engine VS Code uses for hover types.

## Two-level approach

- **Decorators** → read from AST (syntax nodes, no execution needed)
- **Return types + param types** → resolved via Type Checker (`getType()`)

## Normalization output (fed to Mustache)

The parser normalizes everything into this JSON shape before handing to Mustache:

```json
{
  "featureName": "Profile",
  "entityName": "UserProfile",
  "baseUrl": "/api/v1",
  "queries": [
    {
      "name": "getProfile",
      "httpMethod": "GET",
      "path": "/profile/:userId",
      "returnType": "UserProfile",
      "isNullable": false,
      "isArray": false,
      "cacheSeconds": 60,
      "params": [
        { "name": "userId", "type": "string", "decorator": "Param" }
      ]
    }
  ],
  "paginatedQueries": [...],
  "mutations": [...]
}
```

Methods are **pre-split** into `queries`, `paginatedQueries`, and `mutations` arrays so Mustache never needs `if/else` logic.

# 8. Injection — Adding Methods to Existing Files

When a developer adds a new method to the abstract class and reruns `spry build`, Spry **injects** the new method into existing Spry-owned files using **ts-morph's write API**. No regex, no string manipulation — pure AST operations.

## File ownership

**Spry-owned** (always safe to generate or inject):

- All use cases, RepositoryImpl, RemoteDataSource, profileQueries.ts, di.ts, DTOs, mappers, barrel index files

**Developer-owned** (generated once, never touched again):

- `<feature>Store.ts`, domain models, abstract repository

## ts-morph write operations

| Operation | ts-morph method |
| --- | --- |
| Add method to class | `class.addMethod()` |
| Add export to di.ts | `file.addVariableStatement()` |
| Add hook function | `file.addFunction()` |
| Add import | `file.addImportDeclaration()` |

## CLI output when new method is added

```bash
$ spry build profile

✓ Created:  DeleteUserUseCase.ts
✓ Injected: ProfileRepositoryImpl.ts
✓ Injected: ProfileRemoteDataSource.ts
✓ Injected: profileQueries.ts
✓ Injected: di.ts
— Skipped:  profileStore.ts  (developer-owned)
— Skipped:  UserProfile.ts   (developer-owned)
```

# 9. Conflict Detection & Manifest

## `.spry-manifest.json`

Tracks which methods have been generated per feature:

```json
{
  "features": {
    "profile": {
      "generatedMethods": ["getProfile", "updateProfile", "deleteProfile"]
    }
  }
}
```

On `spry build`, Spry diffs the current abstract class methods against the manifest to find new methods → inject only those.

## `.spryrc.json` integrity

On every `spry build`, Spry recomputes the checksum of `.spryrc.json` fields and compares to the stored checksum. If tampered → hard stop:

```bash
✖ Error: .spryrc.json has been manually modified.
  Run `spry init` again to reconfigure.
```

# 10. Error Handling Strategy

Spry generates a complete typed error hierarchy. All errors propagate as typed `DomainError` subclasses — never `unknown`.

## Error classes (generated once in `shared/errors/`)

```tsx
// Base
class DomainError extends Error {
  constructor(public readonly code: string, message: string)
}

// HTTP errors — normalizes fetch AND axios behavior
class HttpError extends DomainError {
  constructor(public readonly status: number, message: string)
  get isNotFound()      // status === 404
  get isUnauthorized()  // status === 401
  get isServerError()   // status >= 500
}

// Network errors — offline, DNS, CORS
class NetworkError extends DomainError {}
```

## Critical: fetch vs axios difference

`fetch` does **NOT** throw on 4xx/5xx — you must check `response.ok`. `axios` throws automatically. Spry's `httpClient.ts` normalizes both into `HttpError` / `NetworkError` so the rest of the architecture never cares which HTTP client was chosen.

## React Query typed errors

```tsx
useQuery<UserProfile, DomainError>({
  queryFn: () => getProfileUseCase.execute(userId),
  retry: (failureCount, error) => {
    if (error instanceof HttpError && error.status < 500) return false
    return failureCount < 2
  },
})
// error is DomainError | null — never unknown
```

## Cache invalidation

Spry generates an empty `onSuccess` with a comment. Cache invalidation is **the developer's responsibility** — Spry ships documentation covering `setQueryData`, `invalidateQueries`, and optimistic update patterns.

# 11. Dependency Injection

Spry generates a `di.ts` file per feature — a singleton module that wires the full dependency chain:

```tsx
// features/profile/di.ts — GENERATED
import { ProfileRemoteDataSource } from './data/datasources/ProfileRemoteDataSource'
import { ProfileRepositoryImpl } from './data/repositories/ProfileRepositoryImpl'
import { GetProfileUseCase } from './domain/usecases/GetProfileUseCase'

const dataSource = new ProfileRemoteDataSource()
const repository = new ProfileRepositoryImpl(dataSource)

export const getProfileUseCase = new GetProfileUseCase(repository)
export const updateProfileUseCase = new UpdateProfileUseCase(repository)
```

Every other file imports from `di.ts`. No heavy DI container (no tsyringe, no inversify).

# 14. Testing Requirements

> **No code is pushed without tests. Every module has a corresponding test file. CI blocks merge on any test failure.**
> 

## Unit tests — `cli/tests/unit/`

Mirror the `src/` structure exactly. One test file per source file. External dependencies mocked.

- **Parser tests** — use fixture repository files, assert normalized JSON output
- **Generator tests** — assert correct template selected, Mustache renders expected output
- **Injector tests** — create in-memory ts-morph Project, inject, assert resulting TypeScript string
- **Manifest tests** — assert read/write/validation against fixture JSON
- **Config tests** — assert checksum generation and tamper detection

## Integration tests — `cli/tests/integration/`

Full CLI commands run in a **temporary directory** per test. No mocking — real file system, real ts-morph, real Mustache.

- `init.test.ts` — runs `spry init`, asserts `.spryrc.json`, tsconfig patches, shared scaffold
- `new.test.ts` — runs `spry new profile`, asserts domain skeleton
- `build.rest.test.ts` — copies fixture repo, runs `spry build profile`, diffs every generated file against `tests/fixtures/expected/rest-fetch/`
- `build.inject.test.ts` — build, add `deleteUser` to abstract class, build again, assert injection without touching developer-owned files
- `build.conflict.test.ts` — asserts `--dry-run` produces no writes, `--force` regenerates

## Test fixtures — `cli/tests/fixtures/`

```
fixtures/
  repositories/
    ProfileRepository.simple.ts       ← 2 methods, basic types
    ProfileRepository.nullable.ts     ← nullable return types
    ProfileRepository.paginated.ts    ← @Paginated decorator

    ProfileRepository.complex.ts      ← deeply nested generics
  spryrc/
    rest.fetch.json

    rest-axios/                       ← expected output files for REST+axios
```

# 16. Key Decisions Log

| Decision | Choice | Reason |  |
| --- | --- | --- | --- |
| Mobile-only | Yes — no RSC | RSC has no meaningful benefit on native, breaks EAS Update |  |
| HTTP client | fetch (default) or axios (option) | fetch = zero deps, axios = interceptors for auth |  |
| Query client | React Query | Best DX, agnostic to HTTP client, typed errors |  |
| State management | Zustand | Simple, zero boilerplate, pairs cleanly with React Query |  |
| GraphQL client | urql (Phase 2) | Lighter than Apollo, better mobile bundle size |  |
| DI pattern | Singleton `di.ts` per feature | Zero deps, simple, testable, Mustache-friendly |  |
| Mustache logic | Pre-split arrays (queries/mutations/paginated) | Avoids any if/else in templates |  |
| File injection | ts-morph write API | AST-aware, no regex, handles all edge cases |  |
| Config protection | Checksum in `.spryrc.json` | Prevents drift, hard stop if tampered |  |
| Cache invalidation | Developer's responsibility | Spry cannot know data relationships, docs cover patterns |  |
| DTO mapping | Mirror of domain model | Developer owns types, they align to their API |  |
| Zustand store | Shell only | Developer fills UI state, Spry handles `create()` wrapper | } |
